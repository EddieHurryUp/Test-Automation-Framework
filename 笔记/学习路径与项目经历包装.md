# Test-Automation-Framework 学习路径与项目经历包装

## 1. 学习目标（先定标准）
- 目标1：能独立新增一个接口测试用例（含参数化、提取、断言、报告）。
- 目标2：能说清框架主链路：`YAML用例 -> 请求执行 -> 结果提取 -> 断言 -> 报告`。
- 目标3：能把该项目包装成有技术深度的项目经历，而不只是“跑过测试”。

## 2. 项目认知地图（建议按这个顺序读）

### 2.1 入口与运行
- `run.py`：测试执行入口，切换 Allure / TMReport。
- `conftest.py`：全局前置与收尾，清理 `extract.yaml` 与报告临时文件。
- `testcase/conftest.py`：会话级登录、每条用例日志切面、预留数据库清理点。

### 2.2 配置层
- `conf/setting.py`：路径、超时、日志级别、报告类型等全局常量。
- `conf/config.ini`：环境 host、数据库配置、报告配置。
- `conf/operationConfig.py`：对 `config.ini` 的读取封装。

### 2.3 用例数据层
- `testcase/**/*.yaml`、`data/loginName.yaml`：测试数据与断言规则。
- `common/readyaml.py`：读取 YAML、拆分 `baseInfo/testCase`、读写 `extract.yaml`。
- `extract.yaml`：跨接口参数关联的“中间状态仓库”。

### 2.4 执行引擎层（核心）
- `base/apiutil.py`：主编排器 `RequestBase.specification_yaml()`。
- `common/sendrequest.py`：统一请求发送、日志与 Allure 附件。
- `common/debugtalk.py`：`${...}` 动态函数（时间戳、提取参数、加密等）。

### 2.5 断言与结果
- `common/assertions.py`：`contains/eq/ne/rv/db` 多断言模式。
- `report/`：历史报告示例（Allure / TMReport）。

## 3. 主链路理解模板（面试必背）
1. `pytest.mark.parametrize` 从 YAML 加载 `baseInfo + testcase`。
2. `RequestBase.specification_yaml()` 解析请求元数据（url/method/header）。
3. `replace_load()` 解析 `${func(args)}` 动态参数（调用 `DebugTalk`）。
4. `SendRequest.run_main()` 执行请求并记录日志/附件。
5. `extract_data/extract_data_list` 将接口返回提取到 `extract.yaml`。
6. `Assertions.assert_result()` 根据 YAML 规则做组合断言。
7. `run.py` 触发 pytest 并生成 Allure/TM 报告。

## 4. 4周系统学习框架（可执行）

### 第1周：跑通与看懂
学习重点：先“跑通”，再“看懂主链路”。

必做任务：
- 配置 `conf/config.ini` 的 `api_envi.host` 指向可访问服务（本地可先用 mock 服务）。
- 执行一次全量：`python run.py`。
- 画一张时序图：从 `test_productList.py` 到 `assert_result()` 的调用流程。

验收标准：
- 你能不用看代码复述 7 步主链路。
- 你知道每一步输入/输出是什么（尤其 `extract.yaml` 的作用）。

产出物：
- 《主链路时序图》
- 《模块职责一页纸》

### 第2周：单接口能力
学习重点：掌握 YAML 用例设计与断言。

必做任务：
- 在 `testcase/Single interface/` 新增 1 个接口 YAML（至少 3 条 case：正向、反向、边界）。
- 至少使用两种断言（例如 `contains + eq`）。
- 为新接口新增一个 pytest 测试文件并参数化运行。

验收标准：
- 新增用例可稳定执行。
- 断言失败时，Allure 报告中能直接看懂失败原因。

产出物：
- 新增 YAML + 测试脚本
- 一份失败案例截图（用于面试讲“问题定位”）

### 第3周：业务流与接口关联
学习重点：掌握跨接口依赖（提取-传递-校验）。

必做任务：
- 复盘 `ProductManager` 场景（商品列表 -> 商品详情 -> 提交订单 -> 支付）。
- 明确每一步提取字段与消费字段（`goodsId/orderNumber/userId`）。
- 自己设计一个新业务流（最少 3 个接口串联）。

验收标准：
- 你能解释 `extract` 与 `extract_list` 的差异和使用场景。
- 你能独立排查“提取失败导致后续接口失败”的问题。

产出物：
- 《业务流依赖表》（字段来源、字段去向、失败影响）
- 1 条完整新业务流用例

### 第4周：工程化与可讲深度
学习重点：把“能用”提升到“可维护、可扩展、可面试”。

必做任务（建议至少做2项）：
- 优化1：给 `replace_load` 增加异常提示（函数不存在、参数不匹配时给出明确错误）。
- 优化2：完善断言错误信息（打印 JSONPath 命中结果与预期差异）。
- 优化3：增加环境分层（`dev/test/prod`）与命令行切换。
- 优化4：把 `extract.yaml` 变为“每次执行唯一文件”，避免并发污染。

验收标准：
- 至少完成 1 个“代码级优化 + 文档说明 + 效果对比”。
- 你能用 3 分钟讲清改动前问题、改动方案、收益。

产出物：
- 1 个优化 PR（或本地提交记录）
- 《优化前后对比说明》

## 5. 每次学习的固定模板（避免低效）
每次 60-90 分钟，按下面节奏：
1. 10分钟：阅读一个模块 + 记录“输入/输出/依赖”。
2. 35分钟：动手改一处（新增 case、改断言、加日志都可以）。
3. 15分钟：跑测试 + 看报告 + 写复盘。
4. 10分钟：沉淀到你的“面试素材库”（问题、方案、结果）。

## 6. 项目经历包装（简历可直接改）

### 6.1 项目名称（推荐）
基于 Pytest + YAML 的接口自动化测试框架（含业务流编排与 Allure 报告）

### 6.2 项目描述（1-2行）
负责搭建并优化接口自动化测试框架，支持 YAML 配置化用例、动态参数替换、跨接口数据提取与多模式断言，覆盖单接口与业务流场景，并输出 Allure 测试报告用于持续回归。

### 6.3 职责与亮点（可直接放简历）
- 设计并实现 YAML 用例驱动机制，统一 `baseInfo/testCase` 结构，降低新增接口用例成本。
- 实现 `${}` 动态函数解析能力，支持时间戳、加密、参数提取等运行时数据组装。
- 构建跨接口参数关联链路（`extract.yaml`），打通登录、下单、支付等业务流自动化。
- 落地 `contains/eq/ne/rv/db` 多断言模式，提升接口结果校验覆盖度与可读性。
- 接入 Allure 报告与 pytest fixture 机制，实现执行前清理、执行后汇总与失败快速定位。

### 6.4 可量化指标（按你实际数据替换）
- 用例规模：累计沉淀 `X` 个接口、`Y` 条自动化用例，覆盖 `Z` 条核心业务链路。
- 执行效率：回归耗时从 `A` 小时下降到 `B` 分钟（或执行效率提升 `N%`）。
- 缺陷收益：提前发现 `M` 个高优先级接口缺陷，降低联调/上线风险。
- 维护效率：新增接口用例平均开发时间从 `C` 分钟降至 `D` 分钟。

### 6.5 面试 STAR 讲法（30秒版）
- S：团队接口回归依赖手工，发布前验证慢且容易漏。
- T：搭建一套可扩展的接口自动化框架，覆盖核心业务流。
- A：基于 Pytest + YAML 实现数据驱动、动态参数解析、接口提取关联和多断言机制，并接入 Allure 报告。
- R：实现核心链路自动回归，显著缩短回归时间，并提升缺陷发现前置率。
